# FFmpeg Magisk Module — Automated Binary Supply (Design)

Date: 2026-06-20
Status: Reviewed & internally consistent (multi-agent written-spec review complete — see §17); ready for implementation
Repo: `ptrkhh/ffmpeg-magisk-module`, branch `main`

## 1. Goal

Turn the current manual skeleton (user must hand-place an `ffmpeg` binary before
flashing) into a module whose arm64 `ffmpeg` + `ffprobe` binaries are supplied
**automatically** from a pinned, verified upstream prebuilt, packaged into a
flashable ZIP by CI, and published as a GitHub Release.

## 2. Scope decisions (locked)

| Decision | Choice |
|---|---|
| Binary supply | Download prebuilt in CI (not build-from-source) |
| Upstream source | `yearsyan/ffmpeg-android-build`, asset `ffmpeg_android_aarch64_gpl.tar.gz` |
| Architecture | arm64-v8a only |
| Binaries shipped | `ffmpeg` + `ffprobe` |
| Codec/license | Full GPL build (libx264 + libx265 encode) |
| Upstream tracking | Pin tag + SHA512; manual bump via `git tag vX.Y.Z && git push --tags`; `workflow_dispatch` is a verify-only dry-run |
| Build entrypoint | Shell scripts (no Makefile) |
| Extras | `updateJson` in-app auto-update, `action.sh` version button, `COPYING` (GPLv2 text) |

### Non-goals (YAGNI)
- Multi-arch (arm/x86/x64). Lock + fetch + `customize.sh` are structured so an
  arch can be added later, but not now.
- `ffplay`.
- Building ffmpeg from source.
- Magisk auto-update is in scope; auto-*publishing* new upstream untested is not.

## 3. Source selection — verification evidence

`yearsyan/ffmpeg-android-build` release `v7.1-beta.16`, asset
`ffmpeg_android_aarch64_gpl.tar.gz`, was downloaded and its `bin/ffmpeg` +
`bin/ffprobe` **executed on a real arm64 Android device** (2026-06-20):

- ffmpeg/ffprobe report version **n7.1.1** (current stable; only the build *tag* is beta).
- `configuration:` includes `--enable-gpl --enable-libx264 --enable-libx265`
  (+ libaom AV1 enc/dec, libdav1d, libmp3lame, MediaCodec HW decoders).
- `--disable-shared --enable-static`: third-party libs are static; the executable
  links **only Android system libs** (`libc/libm/libz/libandroid/libmediandk/libdl`)
  → self-contained, **no `.so` to ship**.
- ELF LOAD alignment `0x4000` → **16 KB page size**, Android 15/16 compatible.
  Note: this is a *per-asset* property of the v7.1-beta.16 binaries; the SHA512 pin
  guarantees byte-identity for that pin only. A future `update-pin.sh` bump to a
  `0x1000`-aligned upstream binary would silently regress 16 KB-device support, so
  the alignment is asserted in CI and at bump time (see §12 step 5 / §11 / MGK-7).
- Toolchain NDK r28b, clang 19, PGO+BOLT+LTO. ~28 MB per binary.
- Distributed as a tagged GitHub Release with downloadable assets → pinnable.
- Internal archive layout (path of `ffmpeg`/`ffprobe` inside the tarball):
  **ASSUMPTION** — the exact internal path (flat `bin/ffmpeg` vs `<prefix>/bin/ffmpeg`)
  is not verifiable in the spec-authoring environment (no network). The extraction
  logic (§6 step 4) is written to tolerate both layouts and hard-fail if neither is
  present; the maintainer confirms the real path with `tar tzf` when first pinning
  and reconciles this section.

Rejected alternatives: `hzw1199` (LGPL, no x264/x265 encode, no tagged releases);
`Khang-NT`/`ivanarh` (2018, NDK r15, no 16 KB support); `arthenica/ffmpeg-kit`
(ships `.aar` libraries, not standalone CLI; prebuilts retired 2025).

**`.gitignore` (exact final content for this path):** replace the single line
`system/bin/ffmpeg` with:
```
system/bin/*
!system/bin/.gitkeep
```
The glob ignores BOTH binaries (and any future extras); the negation keeps the
tracked placeholder so the directory survives a clean checkout (CONS-1/CONS-2).

**Changelog files (CONS-6):** `changelog.txt` is **machine-regenerated per release**
from the annotated git tag message (the tag annotation is the single authoritative
release-note source). `CHANGELOG.md` is the **human-maintained cumulative** repo
history. CI appends the same tag-derived entry to the top of `CHANGELOG.md`. The
first-release seed for `changelog.txt` = the `v7.1.1` tag annotation text.

### Avoid (ianmacd/SQLite3 is the legacy template)
Do **not** introduce `config.sh`, `common/{post-fs-data,service,system.prop}`,
`system/xbin`, or `minMagisk`. Modern `customize.sh` + `system/bin` only.

## 5. `ffmpeg.lock` (supply-chain anchor)

A `KEY=value` data file, single source of truth. It is **parsed by regex, never
sourced** (see §6 step 1 / SEC-8) — it is *data, not code*:

```
UPSTREAM_REPO=yearsyan/ffmpeg-android-build
UPSTREAM_TAG=v7.1-beta.16
FFMPEG_VERSION=7.1.1
ASSET=ffmpeg_android_aarch64_gpl.tar.gz
ASSET_SHA512=<generated by update-pin.sh>
FFMPEG_SHA512=<sha512 of extracted bin/ffmpeg>
FFPROBE_SHA512=<sha512 of extracted bin/ffprobe>
```

SHA512 values are machine-generated by `update-pin.sh`, never hand-typed.

`ffmpeg.lock` is **COMMITTED to git (never gitignored)** — it is the only checked-in
record of the pinned upstream version and SHA512s; CI reads it directly from the
checkout (CONS-5).

**Threat model (cross-ref §13 / SEC-1):** `UPSTREAM_TAG` is a *mutable* git ref and
GitHub release assets can be deleted and re-uploaded under the same filename, so
`ASSET_SHA512` is the **sole immutability anchor**. A SHA512 mismatch means the bytes
served under our pin changed — a STOP condition (do not blindly re-pin to make CI
green); a human must audit upstream first.

## 6. Scripts

### `scripts/fetch-ffmpeg.sh`

Explicit, ordered, **fail-closed** steps (SEC-3):

0. **Host-only script** — `fetch-ffmpeg.sh` runs on the build host (CI `ubuntu-latest`
   / maintainer machine), **not on-device**; write it as portable POSIX `sh` with
   `set -eu`. **FORBID `curl | tar` streaming** regardless of shell: a pipeline would
   extract an unverified ELF *before* the hash is checked, and (absent `pipefail`,
   which POSIX `sh` does not guarantee) a failed download would be silently hidden.
   Download to a temp file, verify, *then* extract.
1. **Parse `ffmpeg.lock` — do NOT `source` it (SEC-8).** Read it line-by-line and
   accept ONLY lines matching `^[A-Z0-9_]+=[A-Za-z0-9._/:+-]+$` (reject/abort on any
   line containing `$`, `` ` ``, `(`, `;`, spaces, or quotes). `UPSTREAM_REPO` from
   the lock MUST equal the allowlisted constant `yearsyan/ffmpeg-android-build`,
   else abort. (To change upstream intentionally, edit the allowlist constant in the
   script in a reviewed commit — not just the lock.)
2. `curl -fL --proto =https --tlsv1.2` the asset
   `https://github.com/$UPSTREAM_REPO/releases/download/$UPSTREAM_TAG/$ASSET`
   to a **temp file** (`$TMPDIR/asset.tar.gz`).
3. **Verify `ASSET_SHA512`** against the temp file
   (`printf '%s  %s' "$ASSET_SHA512" "$tmpfile" | sha512sum -c -`); on non-zero exit,
   `rm` the temp file and `exit 1` — `system/bin/` is left **untouched**. (See §13 /
   §5 threat model: a mismatch is a STOP condition, never an auto-re-pin trigger.)
4. **Extract to a temp dir** (`mkdir -p $TMPDIR/x`). Locate the binaries tolerant of
   an optional top-level wrapper dir: `tar -tzf asset | grep -E '(^|/)bin/ffmpeg$'`
   (and `…/bin/ffprobe$`) and extract **by that exact member name** (handles both
   flat `bin/ffmpeg` and `<prefix>/bin/ffmpeg`). If either member is **not found**,
   ABORT with `expected bin/ffmpeg|bin/ffprobe not present in archive` (CONS-4).
5. **Verify each extracted binary** against `FFMPEG_SHA512`/`FFPROBE_SHA512`; on
   mismatch, `exit 1` leaving `system/bin/` untouched (defense-in-depth).
6. `mkdir -p system/bin`; **atomically `mv`** the two verified binaries into
   `system/bin/`; `chmod 0755`. Idempotent; runnable locally and in CI.

TOFU / provenance controls (SEC-2), additionally in this script when used to take a
*new* pin (i.e. invoked via `update-pin.sh`):
- (a) Query the GitHub API for the release and **ASSERT** `release.author.login` /
  repo owner == `yearsyan` (allowlisted constant); abort otherwise.
- (b) Print a TOFU banner — `WARNING: upstream publishes no signature/checksum; this
  pin trusts the bytes served now. Audit
  https://github.com/yearsyan/ffmpeg-android-build/releases/tag/<tag> before
  continuing.` — and require an interactive `y/N` confirm (bypass only via an
  explicit `--yes` flag the human passes).
- (c) Compare `UPSTREAM_REPO` to the hardcoded allowlist constant (as in step 1).

Note: upstream (`yearsyan/ffmpeg-android-build`) publishes **no GPG signature or
per-asset checksum**, and GitHub artifact attestation is only retrievable for
artifacts built in that repo's own Actions — so verifying a third-party tarball
against an upstream signature/attestation is **unimplementable** (nothing exists to
verify against). The controls above (owner assertion + interactive confirm +
allowlist) are the achievable mitigations.

### `scripts/update-pin.sh <new-tag>`
Fetches the asset for `<new-tag>`, computes all three SHA512s, rewrites
`ffmpeg.lock`, and (optionally) updates `FFMPEG_VERSION` from the binary's
`-version`. Makes a version bump a single command. Additionally:
- Runs the TOFU/owner-assertion + interactive confirm above.
- Runs the **16 KB-alignment assertion** (MGK-7): for each binary, parse
  `readelf -lW system/bin/<bin>` and FAIL if any `PT_LOAD` segment `Align` is
  `< 0x4000` — so a regressed pin is caught at bump time, not just in release CI.
- **Archives the upstream corresponding source** (GPLv2 §3; SEC-6): downloads the
  source the upstream build used (or at minimum the upstream build-script tree at
  that tag) into the repo or release assets, so a later deletion of the upstream tag
  cannot strand GPL compliance.

`ffmpeg.lock` edits are **security-sensitive** and require careful review (§14/§16).

## 7. `customize.sh`

```
- Guard: [ "$ARCH" = "arm64" ] || abort "! arm64-v8a only; this device is $ARCH"
  # $ARCH is the Magisk literal: one of arm/arm64/x86/x64/riscv64. Compare against
  # "arm64" — NOT the ABI string arm64-v8a/aarch64 (those would make this guard
  # always abort). (MGK-3)
- set_perm $MODPATH/system/bin/ffmpeg  0 2000 0755
- set_perm $MODPATH/system/bin/ffprobe 0 2000 0755
- ui_print success + 'ffmpeg'/'ffprobe' now system-wide
```
Drops the current "binary not found" branch — CI guarantees presence.

**Permissions / SELinux note (MGK-4):** `set_perm` signature is
`<target> <owner> <group> <permission> [context]`. No 5th `[context]` arg is passed,
so Magisk applies the default context `u:object_r:system_file:s0`, which permits
system-wide exec. `owner=0 / gid=2000(shell) / mode=0755` follows the `system/bin`
executable convention. (Referenced from §13's exec guarantee.)

## 8. `module.prop`

```
id=ffmpeg_systemwide
name=FFmpeg for arm64-v8a devices
version=v7.1.1
versionCode=202606200
author=ptrkhh
description=Statically built GPL ffmpeg + ffprobe (libx264/libx265) system-wide for arm64-v8a. 16KB-page ready.
updateJson=https://raw.githubusercontent.com/ptrkhh/ffmpeg-magisk-module/main/update.json
```

**Versioning rule (single scheme — CI-1/CI-2/SEC-7/MGK-5/CONS-3):**
`versionCode = YYYYMMDD*10 + R`, where `R` is a **0-based same-day release counter**
(`R=0` default; CI derives `R` by counting existing releases/tags whose date ==
today). Examples: first release 2026-06-20 → `202606200`; a same-day re-release →
`202606201`; the next day → `202606210`. Strictly increasing forever.

- The git release tag = `v$FFMPEG_VERSION` (e.g. `v7.1.1`), matching `module.prop`
  `version` and the ZIP name. The version **STRING is display-only**; Magisk compares
  only the **integer `versionCode`**.
- **Bound:** `versionCode` is always a 9-digit integer
  (max `99991231*10+9 = 999912319 < 2^31-1`), so it is a valid signed-32-bit int.
  **Never zero-pad past 9 digits.**

## 9. In-app auto-update (`updateJson`)

`update.json` (Magisk schema), regenerated by CI on each release:
```
{
  "version": "v7.1.1",
  "versionCode": 202606200,
  "zipUrl": "https://github.com/ptrkhh/ffmpeg-magisk-module/releases/download/<TAG>/FFmpeg-v7.1.1-for-magisk.arm64.zip",
  "changelog": "https://raw.githubusercontent.com/ptrkhh/ffmpeg-magisk-module/main/changelog.txt"
}
```
Magisk app polls `updateJson`, shows "update available", installs the `zipUrl`.
CI commits the refreshed `update.json` + `changelog.txt` back to `main` after publishing.
The `versionCode` here MUST equal the value CI stamps into `module.prop` (§8).

**Integrity limitation (CI-9 / SEC-5):** Magisk's `updateJson` schema has **no
checksum/signature field**; the in-app update path's integrity rests on HTTPS +
GitHub Releases trust only (see §13). This is an **accepted limitation of Magisk**,
not an oversight. The published-ZIP SHA256 (§12 step 6) is for **manual** verification
(published in the changelog/README), not consumed by the Magisk updater.

## 10. `action.sh` (Magisk action button)

Runs on demand from the Magisk app; prints versions as a built-in smoke test.
Binaries are invoked by **absolute path** via `MODDIR` so the test does not depend on
`/system/bin` being on `PATH` or on the overlay mount state in the action context
(BusyBox `ash` Standalone Mode resolves non-applet executables via `PATH`; MGK-2):
```
MODDIR=${0%/*}
"$MODDIR/system/bin/ffmpeg" -version | head -n1
"$MODDIR/system/bin/ffprobe" -version | head -n1
```

**Requires Magisk v28.0+** (`MAGISK_VER_CODE >= 28000`): `action.sh` support was added
in Magisk v28.0. The Action button is a **best-effort extra**. The **install floor
stays at Magisk v20.4** (`update-binary` guard unchanged) — on v20.4–v27.x the module
installs and works system-wide; only the in-app Action button is unavailable (MGK-1).
The **canonical smoke test** is the terminal/adb `ffmpeg -version` / `ffprobe -version`
path (§14/§16); the Action button is *additionally* available on v28.0+.

## 11. `.gitattributes`

```
* text=auto eol=lf
META-INF/** text eol=lf
scripts/**  text eol=lf
*.sh        text eol=lf
module.prop text eol=lf
update.json text eol=lf
changelog.txt text eol=lf
ffmpeg.lock text eol=lf
*.json      text eol=lf
*.txt       text eol=lf
system/**   binary
```
Prevents CRLF from corrupting Magisk shell scripts. The explicit `ffmpeg.lock` /
`*.json` / `*.txt` `eol=lf` rules prevent a CRLF checkout from appending `\r` to each
parsed lock value (which would corrupt the URL / break SHA compares — CI-8). The
`system/** binary` override keeps ELF binaries from being EOL-munged. (Under SEC-8 a
stray `\r` fails the strict regex and aborts fail-closed, but `eol=lf` prevents the
broken checkout entirely.)

The 16 KB-alignment / `readelf` assertions (§12 step 5, `update-pin.sh`) likewise
keep the durable promises of §3.

## 12. CI pipeline (`.github/workflows/release.yml`)

**Triggers:** Releases publish **ONLY on push: `tags: ["v*"]`** (deterministic
tag→Release). `workflow_dispatch` is a **NON-PUBLISHING dry-run**: it runs
checkout→fetch→verify→self-test (steps 1, 2, 5) and STOPS — no Release, no
commit-back (CI-3).

**Workflow-level config (SEC-4/CI-4/CONS-8):**
- `permissions: contents: write` (minimal).
- `concurrency: { group: release, cancel-in-progress: false }` (serialize releases).
- The commit-back (step 7) uses the **automatic `GITHUB_TOKEN`** (never a PAT/deploy
  key) — guaranteeing the commit does **not** re-trigger the workflow (loop
  prevention). `push` triggers MUST NOT include changes to
  `update.json`/`changelog.txt` paths (belt-and-suspenders; `GITHUB_TOKEN` already
  prevents recursion).
- If `main` is branch-protected, the workflow identity must be on the protection
  bypass list; otherwise step 7 **opens a PR** instead of pushing directly.

Steps:

1. Checkout.
2. `scripts/fetch-ffmpeg.sh` (download pinned asset, SHA512 verify fail-closed, extract both binaries).
3. Stamp `module.prop` `version`/`versionCode` from `ffmpeg.lock` (versionCode per §8/§12.3).
3b. **Assert `system/bin` contains exactly the two expected non-dotfiles** — fail
   otherwise (defense-in-depth on the fresh CI checkout; CI-5). The check MUST be
   explicit about the tracked `.gitkeep` (do NOT rely on `ls` hiding dotfiles): e.g.
   `found=$(find system/bin -maxdepth 1 -type f ! -name .gitkeep -printf '%f\n' | sort | tr '\n' ' ')`
   then `test "$found" = "ffmpeg ffprobe "`.
4. Package: run `zip` from the repo root with an explicit include list —
   `zip -r9 FFmpeg-v$FFMPEG_VERSION-for-magisk.arm64.zip META-INF system module.prop customize.sh action.sh COPYING -x '*/.git*' -x 'system/bin/.gitkeep'`
   (`COPYING` ships for GPL compliance — repo/CI files like `scripts`, `docs`,
   `ffmpeg.lock`, workflows are not packaged; the tracked `system/bin/.gitkeep` is
   explicitly excluded so the flashed module carries only the two binaries).
   Fetch-before-package ordering is already correct and is retained.
5. **Self-test** (CI-6/MGK-7): unzip to temp; for each of `system/bin/ffmpeg` +
   `system/bin/ffprobe`:
   - (1) `file <bin> | grep -q 'ELF 64-bit LSB.*ARM aarch64'` (tolerates the
     `pie executable`/`executable` variance across `file(1)` versions);
   - (2) version-stable backstop: `readelf -h <bin>` shows `Class: ELF64` and
     `Machine: AArch64`;
   - (3) `test -x <bin>`;
   - (4) **16 KB alignment:** parse `readelf -lW <bin>` and FAIL if any `PT_LOAD`
     segment `Align` is `< 0x4000` (keeps the §3 16 KB-page promise durable across
     bumps; `binutils`/`readelf` is present on `ubuntu-latest`).
   Fail the build on any miss.
6. **Create GitHub Release for the tag** (only when `github.ref_type == 'tag'`);
   attach the ZIP + its SHA256 sidecar (SHA256 is for **manual** verification — CI-9
   — published to changelog/README; Magisk does not consume it). Also attach the
   **corresponding-source archive** (GPLv2 §3; SEC-6). OPTIONAL: emit
   `actions/attest-build-provenance` for the ZIP as supply-chain metadata (note:
   Magisk cannot consume it; it aids manual audit only — SEC-5).
7. **Regenerate `update.json`** (+ `changelog.txt` from the annotated tag message;
   append the same entry to the top of `CHANGELOG.md`) with the new
   tag/versionCode/zipUrl; commit to `main` using `GITHUB_TOKEN`. Before push:
   `git pull --rebase origin main` then push; on non-fast-forward, retry once.
   (Only when `github.ref_type == 'tag'`.)
   Alternative considered: publish `update.json` as a Release asset and point
   `updateJson` at a stable per-repo raw URL — but since `module.prop` already points
   at `raw .../main/update.json` (§8), the commit-back path is retained and hardened
   as above.

### 12.3 versionCode derivation
`versionCode = $(date -u +%Y%m%d)*10 + R`, where `R` = count of today's prior
releases (0-based). CI MUST **assert `new_versionCode > latest_published_versionCode`**
(read the latest from the most recent `vX` tag/release or `update.json` on `main`);
**abort the release if not strictly greater**. This guarantees a same-day security
re-release always supersedes the bad build (SEC-7).

**Manual bump flow (the sole publish path):** `scripts/update-pin.sh <new-tag>` →
review `ffmpeg.lock` diff → commit → `git tag vX.Y.Z` → push → CI publishes.
Maintainer controls when ffmpeg changes.

## 13. Error handling

- SHA512 mismatch (asset or extracted binary) → hard fail (supply-chain guard).
  **Threat model (SEC-1 / §5):** `UPSTREAM_TAG` is a *mutable* git ref and GitHub
  release assets can be deleted+re-uploaded under the same filename; `ASSET_SHA512`
  is the sole immutability anchor. A SHA512 mismatch means the bytes served under our
  pin **changed** — this is a **STOP condition, not a transient CI failure**. Do NOT
  re-run `update-pin.sh` to make CI green; a human must audit the upstream diff and
  confirm provenance before re-pinning.
- **On ANY fetch/verify failure, `system/bin/` is left untouched** — binaries are
  only moved in after all verifications pass (SEC-3).
- **Asset layout mismatch** (expected `bin/ffmpeg`/`bin/ffprobe` member absent in the
  archive) → hard fail (CONS-4).
- Network / asset-missing → clear failure message.
- Non-arm64 device → `customize.sh` aborts install with a readable message.
- Magisk < v20.4 → existing `update-binary` guard. (Magisk < v28.0 → module still
  installs; only the `action.sh` button is inert — MGK-1, not a failure.)
- Broken/empty ZIP → blocked by CI self-test before any Release is created.
- **Published-ZIP integrity boundary (SEC-5):** the published ZIP's integrity is
  HTTPS + GitHub-Releases-trust only; Magisk `updateJson` has no hash/signature
  field, so **release-write access == code execution on users' devices**. Therefore
  release-write (the `GITHUB_TOKEN` scope and any human with repo write) is the
  security boundary — keep the contributor/maintainer set minimal and require 2FA.
- **Exec guarantee:** binaries install as `0/2000/0755` with default SELinux context
  `u:object_r:system_file:s0` (see §7/MGK-4), which permits system-wide exec.

### 13.1 Rollback (post-publish bad release — CONS-7)
A release that passes the ELF/exec self-test but is runtime-broken cannot be rolled
back by lowering `versionCode` (Magisk treats a lower value as no-update). Procedure:
1. Cut a **NEW** release that re-pins the last-known-good `ffmpeg.lock` (or reverts
   the offending change), with a **strictly-higher** `versionCode` (per §8, e.g.
   `today*10 + R`) so Magisk offers it as an update.
2. Optionally mark the bad GitHub Release as **pre-release** / delete its asset so
   manual installs stop.
3. `update.json` on `main` now points at the good re-release.
Note: users already on the bad build are rescued only by the higher-`versionCode`
re-release, never by deleting the bad one.

## 14. Testing / verification

- CI: ZIP self-test (`file` + `readelf` = arm64 ELF64/AArch64, exec bit, 16 KB
  alignment, both binaries present); `shellcheck` with **`--shell=sh`** (Magisk
  scripts run under BusyBox `ash`; `--shell=sh` flags bashisms that would fail
  on-device) on: `customize.sh`, `action.sh`, `scripts/*.sh`, and
  `META-INF/com/google/android/update-binary` (a real `#!/sbin/sh` script that runs
  first — a syntax error there bricks every install; CI-7). **Do NOT lint
  `ffmpeg.lock`** — per SEC-8 it is regex-parsed data, not sourced shell.
- Local: `scripts/fetch-ffmpeg.sh` then flash on device; **canonical** README smoke
  test `ffmpeg -version` / `ffprobe -version` (terminal/adb); additionally, on Magisk
  v28.0+, the in-app `action.sh` button.
- **`ffmpeg.lock` edits are security-sensitive** and require careful review (SEC-8).

## 15. Licensing

GPL build (libx264 GPLv2+, libx265 GPLv2+) → combined work is **GPLv2-or-later**.
- Ship `COPYING` (GPLv2 text).
- **Corresponding source (GPLv2 §3 — SEC-6):** the obligation to provide complete
  corresponding source falls on the **distributor** (ptrkhh, via the Release ZIP) and
  is **not** discharged by pointing at a third party's mutable tag (SEC-1 establishes
  tags/assets are deletable). The distributor satisfies §3 by **EITHER**:

    archives it and CI attaches it — §6/§12), **OR**
  - (b) including a written **GPLv2 §3(b)** offer (valid 3 years) in `COPYING`/README.
  - **Recommend (a).**
- README "License" section: state the binary is GPL; note `nonfree` is NOT enabled,
  so the build is cleanly redistributable.
- **Trust root (SEC-2):** the prebuilt binary is supplied by a **single third-party
  maintainer** (`yearsyan`); there is **no independent signature**. Users who require
  stronger provenance should build from source per "Build it yourself". README's
